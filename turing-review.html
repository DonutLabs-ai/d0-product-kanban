五、未来规划 — 深度 × 广度 × 完整性

Turing 当前具备三个角色: Context 对齐者、项目经理、Code Reviewer。但距离 Superbot 2.0 目标 (70-80% 工程工作迁移) 还有明确缺口。未来围绕三条轴展开。

5.1 深度 — 现有场景持续优化

已跑通的 8 个场景不是终点, 每个都有下一层可挖。

AI 辅助 PRD → 端到端需求闭环

现状: PRD 完成后和工程执行之间还有断层
  • PRD 完成 → 自动拆解 Linear sub-issues, 附验收标准和涉及模块
  • PRD 变更 → 自动追踪受影响 issue 和测试用例
  • 从 "PRD 到可开工" 进化为 "PRD 到验收完成"

Code Review → 预测性质量门控

现状: 三维度 review 已有, 但事后被动触发
  • 理解 PR 依赖图, 建议最优合并顺序
  • 积累 review 历史 → 识别高风险模块和反复出问题的 pattern
  • 从 "这个 PR 有问题" → "这个模块最近变动频繁, 要重点看"

成本分析 → 持续监控 + 自动预警

现状: 人工触发的一次性分析
  • 定时 cron 拉取 sub2api / AWS / PostHog 成本, 异常自动推送
  • 每周趋势报告, 按用户/cron/场景拆分
  • 从 "发现问题" → "问题发生前预警"

Scanner → 智能项目管理

现状: 每 2h 扫描 + 预警, 只做检测不做干预
  • 阻塞时自动分析原因 (缺 assignee? 等上游? 技术卡点?) 并建议解法
  • Sprint 结束前自动风险评估: 哪些 issue 可能 slip
  • 从 "告诉你有问题" → "告诉你怎么解决"

5.2 广度 — 新场景覆盖

8 个场景覆盖了产品/工程/安全/成本/战略/GTM/HR/效率。空白区域:

QA / 测试自动化

  • PR 提交 → 自动生成/更新测试用例 → 跑测试 → 报告覆盖率
  • 关联: PRODUCT-2243 (封闭式验证环境)

Onboarding / 知识传承

  • 新成员加入 → 基于 memory 自动生成个性化 onboarding 路径
  • "去问 Turing" 成为新人第一反应

运维 / 线上问题响应

  • 告警 → 自动拉 trace + logs → 定位根因 → 建议修复
  • alert-aggregator + infra-debug skill 已有, 需打通自动触发

用户行为洞察

  • 定期扫描 PostHog 漏斗 → 发现 drop-off → 自动创建产品 issue

5.3 完整性 — 闭环是核心

深度和广度是渐进优化, 完整性是质变。Turing 当前最大缺口: 没有闭环。能读能想能建议, 但不能独立完成从想法到交付的全流程。

角色当前目标
Context 对齐者✅ 15 人同时沟通, 自动沉淀→ 主动推送 context 给需要的人
项目经理✅ 扫描 + 预警 + 标准化→ 预测风险 + 建议解法
Code Reviewer✅ 三维度 review→ 预测性质量门控
Researcher⚡ 读 + 分析 + 沉淀→ 研究 → 实验 → 验证 → 提 PR
Engineer❌ 能 review 不能交付→ 封闭式 feature + 测试 + 验证

现在能: 读代码 → review → 定位 bug → 技术方案。不能: 写代码 → 测试 → 验证 → 提交可合并 PR。

自进化: 让系统自己变好 — PRODUCT-2245

四层 Loop, 频率递减价值递增:

  • Daily: Channel digest + 团队分享 → 自动更新 memory 和 skill
  • Weekly: Changelog + 管理 insight → 反馈到组织架构和模块归属
  • Task Feedback: 对话中的抱怨/建议 → 分析 skill/memory 匹配度
  • Spec 自进化: 从上述三层提取最精华 → 更新 SOUL.md / AGENTS.md
